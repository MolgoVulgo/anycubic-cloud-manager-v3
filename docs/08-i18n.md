# Internationalization

Status: `PARTIAL`.

## Goal

The application must support English and French without mixing untranslated UI text, technical logs and user-facing messages. English is the default documentation language. French documentation and UI translations remain first-class project outputs.

## Existing base

The project already has Qt translation infrastructure and language catalogs. The remaining work is not only technical; it is classification and consistency.

## Text categories

| Category | Meaning | Treatment |
| --- | --- | --- |
| A | Static UI text | Must use translation functions/catalogs. |
| B | User-facing dynamic message | Must use translatable templates with parameters. |
| C | Model labels visible to users | Must be localized or mapped to localized display labels. |
| D | Technical/debug text | Can remain technical, especially in debug-only panes. |
| E | Ambiguous text | Must be classified before modification. |

## Architecture rules

- QML visible strings must be translatable.
- Backend must not concatenate final visible messages from hardcoded fragments.
- Dynamic values must be parameters, not string concatenation.
- Logs are not UI messages.
- Debug payloads can stay in technical English if they are not user guidance.
- Error codes and raw protocol keys remain stable technical identifiers.

## Migration lots

1. Static QML labels.
2. QML-visible statuses.
3. Model labels and list entries.
4. Backend messages visible to users.
5. Terminology and catalogs.
6. Regression tests and screenshots/manual checks.

## Validation

A change is i18n-compliant when:

- no new visible text bypasses translation;
- French accents are preserved correctly in UTF-8;
- dynamic values use placeholders;
- technical logs remain separate from user messages;
- English and French labels keep the same functional meaning.

## Decision

i18n is not a final polishing task. It is part of the UI contract. Any new production UI text must be classified and translated at the time it is introduced.
